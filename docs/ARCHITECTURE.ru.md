# Архитектура центрального Planning Lite

## Два разных жизненных цикла

```text
Центральный репозиторий
  prompts / skills / control / templates / defaults
                 |
                 | Git tag + Copier update
                 v
Рабочий проект
  project goal / plans / recommendations / decisions / progress
```

Центральные файлы описывают **как вести проекты**. Локальные файлы описывают **что происходит в конкретном проекте**.

## Почему не submodule

В рабочем проекте нет вложенного `.git`, отдельного указателя на commit submodule и дополнительного шага `git submodule update`. Все установленные файлы являются обычной частью истории рабочего проекта.

## Почему Copier

Copier хранит путь к шаблону и установленную версию в отдельном answers-файле, а при обновлении сравнивает старый шаблон, новый шаблон и эволюцию рабочего проекта. Если участок нельзя объединить безопасно, он создаёт обычный merge conflict вместо тихого уничтожения локальной правки.

## Три категории файлов

### Managed

Можно редактировать только в центральном репозитории. Если их изменить локально, обновление может потребовать ручного merge.

### Project-owned

Создаются один раз и затем пропускаются через `_skip_if_exists`. Они хранят цели, планы, решения и фактическое состояние проекта.

### Installer metadata

`.copier-answers.planning-lite.yml` отслеживается Git, но вручную не редактируется.

## AGENTS.md

`AGENTS.md` принадлежит проекту, потому что в существующем репозитории уже могут быть свои инструкции. Planning Lite добавляет только отмеченный bridge-блок, который отправляет агента к управляемому `.planning/control/ROOT_ROUTER.md`.

Это позволяет централизованно менять маршрутизацию режимов, не перезаписывая локальные инструкции корневого файла.

## Конфигурация

```text
framework/defaults.yml   централизованно обновляется
CONFIG.yml               содержит только локальные overrides
```

Эффективная конфигурация получается рекурсивным merge.

## Версионирование

Git tag является единственным источником релизной версии:

- `hatch-vcs` передаёт версию в metadata Python-пакета;
- `planning_lite.__version__` читает установленную metadata через `importlib.metadata`;
- установленная версия шаблона хранится Copier в `.copier-answers.planning-lite.yml`;
- отдельный `.planning/VERSION` не используется.

Правила номера выпуска:

- Patch: формулировки, исправления ссылок, безопасные уточнения.
- Minor: новый Skill, режим, шаблон или совместимое правило.
- Major: несовместимая структура каталогов, переименование ключевых документов, изменение модели состояния.

Релиз создаётся командой `planning-lite release patch|minor|major`. Она проверяет репозиторий, завершает раздел `Unreleased` в changelog, создаёт release-коммит и annotated tag, но не выполняет push.
