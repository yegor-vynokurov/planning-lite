# Project completion criteria

Status: `Unbaselined`
Last reviewed: `YYYY-MM-DD`

This document defines how the whole project will be judged complete or successful. It is not the same as the Definition of Done for one code change.

## Completion rule

The project may be called complete only when:

1. all criteria marked `Must` are satisfied with recorded evidence;
2. no unresolved blocker contradicts the stated purpose;
3. residual limitations and accepted risks are documented;
4. the intended user or decision-maker can perform the core workflow;
5. the project can be operated, reproduced, or handed over at the level promised in the charter.

Adapt this rule to the project. A continuously evolving product may use milestone completion rather than permanent final completion.

Recommended categories to consider:

- functional outcomes;
- data correctness and integrity;
- reliability and recovery;
- performance and resource limits;
- usability and interfaces;
- testing and reproducibility;
- documentation and onboarding;
- deployment and operations;
- security, privacy, and licensing;
- maintenance or handover.

## Unresolved questions that prevent honest progress estimation

- Is the intended user clear?
- Is the core workflow clear?
- Is there an agreed first useful milestone?
- Are quality thresholds measurable?
- Is deployment or handover part of completion?

## Progress reporting rule

Do not report a single completion percentage unless:

- the target criteria are baselined;
- the scoring scale is defined;
- weights are explicit;
- unknown criteria are not silently treated as zero or complete;
- the evidence and confidence are shown.

When the target is vague, report `Not currently scorable` and describe what must be clarified. A provisional range may be used only with explicit assumptions.
