# Refine project goal and completion criteria

## Authorization boundary

Update project-definition documents only. Do not modify code or create implementation tasks.

## Goal

Turn a vague aspiration into a concise project charter and measurable completion criteria without pretending unresolved product decisions are settled.

## Procedure

1. Read current charter, completion criteria, assessment, recommendations, README, and user-provided intent.
2. Distinguish:
   - problem and desired outcome;
   - users and stakeholders;
   - functional scope;
   - non-goals;
   - architecture quality goals;
   - constraints;
   - project completion;
   - completion of one change.
3. Draft the smallest accurate one-sentence purpose.
4. Define a first useful milestone if the project is open-ended.
5. Create measurable criteria with IDs, thresholds, verification methods, and evidence locations.
6. Mark unresolved decisions explicitly and create recommendation items when clarification work is needed.
7. Update `PROJECT_CHARTER.md`, `PROJECT_COMPLETION_CRITERIA.md`, and `CURRENT_STATE.md`.
8. Explain which criteria are now scorable and which remain provisional.

## Quality bar

Avoid criteria such as "works well", "good quality", "user friendly", or "complete documentation" unless concrete scenarios or thresholds define them.
